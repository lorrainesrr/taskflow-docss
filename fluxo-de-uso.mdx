# Basic Usage Flow

TaskFlow has a mandatory order of operations. Understanding this before you start will prevent most errors.

## The order you need to follow
Each step depends on the previous one. There are no shortcuts.

## Why this order exists

The API was designed to reflect how work actually happens: tasks always belong to a project. A `task_id` only exists after the project has been created and the task has been added to it.

## Task statuses

Every task starts with the status `pending`. From there, you can move it forward as work progresses:

| Status | Meaning |
|---|---|
| `pending` | The task has not been started |
| `in_progress` | The task is actively being worked on |
| `done` | The task has been completed |

You update the status using the `PATCH /tasks/{task_id}` endpoint.

## Endpoints in order of use

| Order | Method | Endpoint | What it does |
|---|---|---|---|
| 1 | POST | `/projects` | Creates a project |
| 2 | POST | `/tasks` | Creates a task in the project |
| 3 | GET | `/projects/{project_id}/tasks` | Lists tasks in the project |
| 4 | PATCH | `/tasks/{task_id}` | Updates a task |
| 5 | DELETE | `/tasks/{task_id}` | Deletes a task |

For a full example with requests and responses, see the [Guide: from start to finish](/guia-tarefas).
