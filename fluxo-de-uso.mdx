# Fluxo básico de uso

A TaskFlow tem uma ordem obrigatória de operações. Entender isso antes de começar evita a maioria dos erros.

## A ordem que você precisa seguir

```
1. Criar um projeto        → você recebe um project_id
2. Criar uma tarefa        → você usa o project_id e recebe um task_id
3. Atualizar ou apagar     → você usa o task_id
```

Cada etapa depende da anterior. Não existe atalho.

## Por que essa ordem existe

A API foi desenhada para refletir como o trabalho funciona na prática: tarefas sempre pertencem a um projeto. Um `task_id` só existe depois que o projeto foi criado e a tarefa foi adicionada a ele.

## Os status de uma tarefa

Toda tarefa criada começa com o status `pending`. A partir daí, você pode avançar conforme o trabalho progride:

| Status | Significado |
|---|---|
| `pending` | A tarefa ainda não foi iniciada |
| `in_progress` | A tarefa está sendo trabalhada |
| `done` | A tarefa foi concluída |

Você atualiza o status usando o endpoint `PATCH /tasks/{task_id}`.

## Resumo dos endpoints por ordem de uso

| Ordem | Método | Endpoint | O que faz |
|---|---|---|---|
| 1 | POST | `/projects` | Cria um projeto |
| 2 | POST | `/tasks` | Cria uma tarefa no projeto |
| 3 | GET | `/projects/{project_id}/tasks` | Lista as tarefas do projeto |
| 4 | PATCH | `/tasks/{task_id}` | Atualiza uma tarefa |
| 5 | DELETE | `/tasks/{task_id}` | Apaga uma tarefa |

Para um exemplo completo com requisições e respostas, veja o [Guia: do início ao fim](/guia-tarefas).
